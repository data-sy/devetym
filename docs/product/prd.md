# 앱 이름 결정 배경

- **표시 이름**: 개발 어원 사전
- **번들 ID**: `com.robin.devetym`

표시 이름 "개발 어원 사전"은 한국어 타겟 사용자가 앱을 직관적으로 이해하기 위한 이름이다. 실제로 이 앱이 하는 일의 본질은 **Etymology(어원 연구)**이고, 번들 ID의 `etym`은 그 근본 개념을 반영한다.

이름 결정 과정:
- `DevEtym` → "Etymology"가 한국어 사용자에게 낯설어서 (표시 이름에서) 제외
- `개발 어원 사전` → 타겟에게 직관적, 스토어 검색 친화적
- 번들 ID는 `devetym`으로 고정 — 나중에 표시 이름을 바꿔도 호환
- 리브랜딩 후보: `Rootly` (추후 검토)

# 문제 정의

개발 공부를 시작한 주니어 개발자와 직무 전환자는 영어 기술 용어가 낯설다. 용어의 뜻은 어딘가에서 찾을 수 있지만, **왜 그 이름인지**는 어디에도 없다. 이름의 유래를 모르면 비슷한 용어끼리 헷갈리고, 개념이 머릿속에 붙지 않는다.

---

# 타겟 유저

**주요 타겟**: 한국어를 사용하는 주니어 개발자 및 직무 전환자
- 강의나 문서를 보다가 용어가 나왔을 때 빠르게 검색하고 싶다
- 영어 용어가 낯설고, 어원을 알면 이해가 쉬워진다고 느끼는 사람
- 면접 준비 중 CS 용어 정리가 필요한 사람

Android와 iOS를 모두 지원하는 이유: 타겟(주니어·전환자)의 단말이 두 플랫폼에 고르게 퍼져 있고, 한 코드베이스로 양쪽에 동시에 닿는 게 개인 개발 규모에서 합리적이기 때문이다.

# 유저 스토리

## US-01: 용어 검색
> 나는 **강의 중 모르는 용어가 나왔을 때**, **빠르게 검색해서 어원과 작명 이유를 확인하고 싶다**, **그래서 용어를 개념과 함께 이해할 수 있다**.

**인수 기준**
- 검색창에 영어 용어를 입력하면 결과가 즉시 표시된다
- 결과에는 어원, 작명 이유, 한 줄 요약이 포함된다
- 번들 DB에 있는 용어(650개)는 오프라인 즉답, DB에 없으면 AI가 생성한 설명을 보여준다 (폴백)
- AI 생성은 로딩 표시 후 결과 표시

## US-02: 검색 히스토리
> 나는 **예전에 찾아봤던 용어를 다시 까먹었을 때**, **히스토리에서 바로 찾아보고 싶다**, **그래서 다시 검색하는 수고를 덜 수 있다**.

**인수 기준**
- 검색한 용어는 자동으로 히스토리에 저장된다
- 히스토리는 최근 검색순으로 표시된다
- 히스토리 항목을 탭하면 바로 해당 용어 상세 화면으로 이동한다
- 개별 항목 삭제 및 전체 삭제가 가능하다

## US-03: 용어 북마크
> 나는 **나중에 다시 보고 싶은 용어를 저장하고 싶다**, **그래서 면접 준비나 복습 때 빠르게 모아볼 수 있다**.

**인수 기준**
- 상세 화면에서 북마크 버튼으로 저장/해제할 수 있다
- 북마크 목록은 별도 탭에서 확인할 수 있다
- 북마크는 앱을 닫아도 유지된다 (로컬 저장)

# 콘텐츠 구조 (용어 카드)

| 항목 | 설명 | 예시 (mutex) |
|---|---|---|
| **용어** | 검색한 영어 용어 | mutex |
| **한 줄 요약** | 핵심만 담은 한국어 한 문장 | 동시 접근을 막는 잠금 장치 |
| **어원** | 단어의 어원 분해 | mutual + exclusion → 상호 배제 |
| **작명 이유** | 왜 이 이름이 붙었는지 | 두 스레드가 '서로(mutual) 막는(exclusion)' 개념 |

---

# 콘텐츠 전략

**큐레이션 DB + AI 폴백 방식**
- 자주 검색될 핵심 용어를 배치로 사전 생성해 **번들 DB(650개)**로 제공 → 오프라인 즉답
- DB에 없는 용어는 Claude로 실시간 생성 (온라인)
- AI 실시간 생성 결과는 로컬에 캐시해서 이후 동일 검색 시 재사용 (비용 절감)

**콘텐츠 신뢰도 고지 방식**
- 모든 콘텐츠는 AI 생성 (배치 DB 포함, 사람 검수 없음)
- AI 생성 뱃지 사용 안 함 (모든 콘텐츠가 동일 조건이므로 구분 불필요)
- **온보딩 화면**에서 1회 고지: "이 앱의 모든 설명은 AI가 생성합니다. 오류가 있을 수 있으니 제보해 주세요."
- **결과 화면 하단**에 고정 링크: "오류 제보하기"

# 화면 구성 (MVP)

탭바 3탭(하단 고정): **검색 / 북마크 / 히스토리**. 결과 화면은 push 이동하되 탭바는 유지.

- **검색창은 화면 하단**에 둔다 — 한 손 엄지 도달성 우선.
- 네비게이션은 Compose Navigation 패턴을 따른다.
- 다크모드는 시스템 설정에 자동 대응한다.

> 시각 결과물(색·폰트·간격)은 각 플랫폼 관례와 디자인 판단에 따라 달라질 수 있다. PRD는 *무엇을 보여주는가*까지만 규정하고, *어떻게 보이는가*는 디자인 문서 소관이다.

# 기술 스택

- **UI 프레임워크**: Compose Multiplatform (Android + iOS 공유)
- **언어**: Kotlin Multiplatform
- **상태 관리**: ViewModel + `StateFlow` (단방향 흐름)
- **데이터 저장**: 로컬 DB (히스토리·북마크·AI 캐시) — SQLDelight 또는 Room KMP, [ADR에서 확정](../adr/)
- **네트워크**: Ktor Client + `kotlinx.serialization` (엔진: Android=OkHttp, iOS=Darwin)
- **DI**: Koin
- **AI 연동**: Claude (폴백 생성용, Cloudflare Worker 프록시 경유 — 기기당 일 10회)
- **큐레이션 DB**: 앱 번들 내 JSON 파일 (`terms.json`, 650개)
