## 1. 제품 개요

**DevEtym (개발 어원 사전)** — iOS 18+ 네이티브 앱. 개발 용어 하나를 입력하면 그 용어의 **어원**과 **왜 그 이름이 붙었는지**를 한국어로 설명한다.

- **독자**: 한국 개발자. 라틴어·그리스어 등 어원 배경지식 없음 가정.
- **입력**: 영문 개발 용어 (예: `mutex`, `idempotent`, `daemon`). 한글·오타·비개발어도 들어올 수 있음.
- **출력 3분기**:
  1. 개발 용어로 판단되면 → 상세 결과 (아래 필드)
  2. 개발 용어 아니면 → "개발 용어를 검색해주세요" 안내
  3. 오타로 추정되면 → "xxx을(를) 찾으셨나요?" + 추천 버튼

**상세 결과 필드 (모두 필수)**
- `keyword` — 영문 소문자
- `aliases` — 한글 음차·풀네임·철자 변이 등 대체 표기 배열 (정의/번역 금지)
- `category` — 6개 고정값 중 하나: `동시성`, `자료구조`, `네트워크`, `DB`, `패턴`, `기타`
- `summary` — 20~30자, 개념 한 줄 요약
- `etymology` — 60~120자, 원어와 뜻·구성 요소
- `namingReason` — 150~270자, **"어원 의미 → 개발 현장 쓰임"으로 다리를 놓는 핵심 필드**

## 2. 기술 스택·API 제약

- **모델**: `claude-sonnet-4-6` (Anthropic Messages API)
- **Extended thinking**: `thinking: {type: "enabled", budget_tokens: 2000}` 활성화. 모델이 응답 전 내부 추론 가능
- **Tool Use**: 세 도구 정의 + `tool_choice: auto`로 구조화 응답 강제
- **Prompt caching**: system 블록에 `cache_control: {type: "ephemeral"}`. tools도 함께 캐시됨

**중요한 API 제약 (회피 불가)**
- Extended thinking과 `tool_choice: any`/`tool`(forcing)은 **공존 불가** — Anthropic API가 400으로 거부함.
  → 그래서 `tool_choice: auto`를 쓰고, 프롬프트에서 "반드시 도구 호출, 텍스트 응답 금지"를 강하게 지시해서 구조를 보장한다.
- `max_tokens: 4096` (thinking 2000 + 응답 2000+ 여유)
- 카테고리는 tool schema의 `enum`으로도 고정되어 있어서, 프롬프트에서 자유 서술한다고 바뀌지 않음

## 4. 현재 시스템 프롬프트 (전문, v1 적용 후)

```

[독자와 목표]
- 독자는 한국 개발자이며, 라틴어·그리스어 등 어원 배경지식이 없다고 가정합니다.
- 어원을 나열하는 것이 아니라, "그 어원이 왜 이 개발 개념의 이름이 되었는가"를 납득시키는 것이 목표입니다.

[도구 선택 — 매우 중요]
- 당신은 모든 입력에 대해 반드시 세 도구 중 정확히 하나를 호출하여 응답해야 합니다. 일반 텍스트 응답은 절대 허용되지 않습니다.
- 입력이 개발 용어로 판단되면 return_term_entry 도구를 호출하여 각 필드를 채우세요.
- 입력이 개발 용어가 아니면 return_not_dev_term 도구를 호출하세요.
- 입력이 개발 용어의 오타로 추정되면 return_possible_typo 도구를 호출하고 suggestion에 올바른 용어를 넣으세요.

[필드별 작성 기준 — return_term_entry]
- aliases: 동일 개념을 지칭하는 대체 표기의 배열. 허용:
  (1) 한글 음차 (예: "뮤텍스", "데몬")
  (2) 약어의 풀네임 (예: "Java Persistence API")
  (3) 철자 변이 (예: "demon")
  정의·번역·상위 개념은 포함하지 않는다 (예: "소프트웨어 결함"은 alias가 아님).
  보통 1~3개. 적절한 대체 표기가 없으면 빈 배열을 반환한다.
- summary: 20~30자. 개념을 한 줄로 요약. "무엇을 하는/무엇인" 수준.
- etymology: 60~120자. 원어(언어·원형)와 그 뜻, 구성 요소(어근·접두사)를 서술.
- namingReason: 150~270자. 반드시 "어원상의 의미 → 개발 현장에서의 실제 쓰임"으로 다리를 놓을 것. 최초 등장 시점·명명자 등 역사적 맥락이 있으면 함께 기술.
- 톤: 건조하고 정확하게. "~이다", "~을 뜻한다" 같은 서술형. 감탄사·과장된 형용사·수식어 남발 금지.

[정확성 원칙]
- 어원이 불확실한 경우 etymology 서두를 "정확한 어원은 불분명하나"로 시작하여 알려진 설만 서술하세요.
- 추측이나 민간어원(folk etymology)을 사실처럼 단정하지 마세요.
- 약어(acronym)의 경우 반드시 각 글자가 무엇의 약자인지 풀어서 명시하세요.

[카테고리 규칙]
- category 값은 스키마의 enum에 명시된 6개 중 하나여야 합니다.
- 6개 분류에 애매하게 걸치는 경우 가장 핵심적인 분류를 선택하세요.
- 어느 분류에도 명확히 속하지 않으면 "기타"를 사용하세요.

## 5. 현재 도구 정의 (요약)

모델이 호출할 수 있는 세 가지 도구. 각 도구의 `description`과 필드 `description`도 프롬프트의 일부로 모델에 전달됨을 기억.

### `return_term_entry`
- **설명**: "입력이 개발 용어로 판단될 때 호출합니다. 어원과 작명 이유를 각 필드에 채워 반환합니다."
- **필드** (모두 required):
  - `keyword` (string) — "영문 소문자 표기. 입력이 한글이거나 대문자여도 정규화하여 넣습니다."
  - `aliases` (array<string>) — "한글 표기나 풀네임 등 대체 이름의 배열"
  - `category` (enum) — `["동시성", "자료구조", "네트워크", "DB", "패턴", "기타"]`
  - `summary` (string) — "20~30자 분량의 한 줄 요약"
  - `etymology` (string) — "60~120자 분량. 원어(언어·원형)와 뜻, 구성 요소(어근·접두사)를 서술."
  - `namingReason` (string) — "150~270자 분량. 어원상 의미와 개발 현장에서의 실제 쓰임 사이에 다리를 놓는 설명."

### `return_not_dev_term`
- **설명**: "입력이 개발 용어가 아닐 때 호출합니다. 입력값은 없습니다."

### `return_possible_typo`
- **설명**: "입력이 개발 용어가 아니지만 개발 용어의 오타로 추정될 때 호출합니다."
- **필드**: `suggestion` (string, required) — "오타를 교정한 올바른 개발 용어"

## 6. 실측 응답 샘플 (v1 적용 후, 2026-04-30 측정)

`Scripts/probe_prompt.py`로 5개 keyword를 한 세션에서 연속 호출. 첫 호출만 cache_create, 나머지 4개는 cache_read 히트 확인. 모델·thinking·tools·tool_choice·system은 iOS `ClaudeAPIService.swift`와 동일.

### 길이 준수 요약

| keyword | summary (20~30) | etymology (60~120) | namingReason (150~270) |
|---|---|---|---|
| mutex | 29자 ✓ | **136자 ✗ (+16)** | 238자 ✓ |
| idempotent | 26자 ✓ | 100자 ✓ | **292자 ✗ (+22)** |
| bug | 21자 ✓ | 122자 ✗ (+2) | 226자 ✓ |
| cookie | 26자 ✓ | 107자 ✓ | **294자 ✗ (+24)** |
| ping | 30자 ✓ | 95자 ✓ | **292자 ✗ (+22)** |

- summary 5/5 준수
- etymology 3/5 준수 (mutex +16자가 가장 큼, bug +2자 미세)
- namingReason 2/5 준수 — 초과 3건 모두 +22~24자로 일관된 폭
- v1에서 namingReason 상한을 300→270으로 낮췄지만, 모델이 여전히 ~290자로 수렴. **상한을 target으로 보는 경향**이 v1 라운드 가설대로 재현됨

### Input: `mutex`
```
category: 동시성
aliases: ["뮤텍스", "mutual exclusion"]

## 7. 개선하고 싶은 방향 — 측정 기반 가설

> 아래는 §6 측정에서 도출한 **가설**입니다. 단정이 아니라 전문가에게 검증·반박을 받기 위한 토론 시드. 데이터 없는 항목은 "측정 안 됨"으로 표시.

### 7.1. 길이 준수율 — 가장 큰 이슈로 보임
- **가설 A (강한 신호)**: `namingReason` 상한이 모델에게 *target*으로 작동. v1에서 300→270으로 낮춘 직후에도 측정 3건이 +22~24자로 **일관된 폭으로 초과**(idempotent 292, cookie 294, ping 292). 새 상한을 새 target으로 학습한 것으로 보임
- **추론**: 단순 상한 추가 인하(예: 250자)는 또 같은 패턴을 만들 가능성. 다른 메커니즘이 필요할 수 있음 — 예: "마지막 문장이 재진술·요약이면 생략할 것" 같은 명시 지시, 또는 "최대 N자 이내, 그러나 짧을수록 좋다"식 비대칭 압박
- **가설 B**: `etymology`도 mutex 케이스에서 +16자 초과. few-shot 예시(mutex)와 동일 keyword인데도 "포트만토 형태" 같은 새 정보를 덧붙임. 정보 가치는 있으나 길이 비용 동반
- **전문가에게**: 어느 접근이 효과적인가 — 상한 추가 인하 / 명시 지시 추가 / Extended thinking 셀프-에디트 유도 / 다른 메커니즘?

### 7.2. Extended thinking 활용
- v1에서 명시적 비변경. 길이 문제가 thinking 단계의 셀프-에디트로 해결될 여지가 있음 — "thinking 안에서 초안을 쓰고, 길이를 세어보고, 잘라낼 것"을 명시 유도하면 효과 있을지
- 그러나 thinking budget 2000토큰의 효과 대비 프롬프트 길이 증가 비용이 정당화되는지 판단 필요

### 7.3. Few-shot 예시 커버리지·자기참조 문제
- **가설**: few-shot에 포함된 keyword(`mutex`/`jpa`/`daemon`/`bug`)를 다시 검색하면 모델이 거의 그대로 베낌. bug 측정에서 etymology 122자·namingReason 226자가 few-shot 예시값과 사실상 동일 — 정상 동작이지만 **회귀 테스트 가치가 낮음**을 의미
- **함의**: 실제 신규 keyword에서의 품질을 측정하려면 few-shot **외부** keyword로 회귀를 봐야 함. sanity test가 보장하는 것은 "프롬프트에 키워드가 박혀 있다"뿐
- **추가 질문**: 5번째 예시가 가장 임팩트가 클 케이스는? (현재 빈틈: 비라틴계 어원, 약어가 아닌 단축형, 인명에서 유래한 용어 등)

### 7.4. 카테고리 6개 분류 마찰
- **가설**: idempotent 측정에서 category가 `네트워크`로 분류됨. 멱등성은 분산시스템·API 일반 개념이라 `패턴` 또는 `기타`도 타당하나, 모델이 HTTP 메서드 사례에 끌려간 듯
- **추론**: 6개 enum이 좁아 모델이 "가장 가까운" 분류로 강제 수렴. 프롬프트에 분류 기준 축(구현 영역 vs 추상 개념 vs 도구류)을 명시할지, enum 자체를 손볼지(스키마 변경은 코스트 큼)

### 7.5. Aliases 스코프 회색지대
- **가설**: v1에서 추가한 스코프 규칙(한글음차/약어풀네임/철자변이)에도 불구하고 cookie의 aliases가 `["쿠키", "HTTP cookie", "웹 쿠키"]`로 나옴. "웹 쿠키"는 번역·한정 표현에 가까워 규칙 위반에 근접
- **추론**: 규칙이 있지만 모델 적용이 느슨. "한정 표현(`웹 X`, `HTTP X`)은 alias 아님" 같은 negative example 추가가 도움이 될지

### 7.6. `return_not_dev_term` / `return_possible_typo` 분기
- §6에서 측정 안 됨. 별도 측정 라운드가 필요함을 인지
- 알려진 회색 지대: `react`(영어 일반어이자 라이브러리), `swift`(형용사이자 언어), `go`(동사이자 언어) 같은 다의어 — 어느 분기로 갈지 모호

### 7.7. 톤 일관성
- §6 5개 응답 모두 "~이다/~다" 서술형 유지. 큰 이슈 없어 보임
- 단 ping의 마지막 "Muuss 본인도 …밝혔다" 같이 살짝 부언 톤이 들어감 — 이게 일반적 패턴인지 산발인지는 더 측정 필요

### 7.8. 약어 풀이 / 비영어 어원
- 약어 풀이: 측정 안 됨 (JPA few-shot 외 신규 두문자어 미측정)
- 비라틴/그리스 어원: 측정 안 됨 (게르만·아랍·일본어 등 케이스 미측정)
- 둘 다 다음 측정 라운드 후보

## 8. 전문가와 대화할 때 고려할 제약

- **구조 변경은 어려움**: 카테고리 6개·필드 세 개(summary/etymology/namingReason)·길이 범위는 UI와 DB 스키마가 참조하는 계약. **프롬프트·도구 description 내부의 품질**을 개선하는 방향이 현실적
- **Few-shot 예시 교체는 가능**하지만 sanity test(`test_systemPrompt_containsFewShotExamples`)가 `mutex`, `jpa|JPA`, `daemon`, `bug` 네 keyword를 기대함. 교체 시 테스트도 수정
- **drift 테스트 존재**: `test_fieldLengthConstraints_matchBetweenPromptAndToolSchema`가 프롬프트 본문과 tool schema의 길이 기준을 정규식으로 비교. 한쪽만 바꾸면 fail
- **캐시 히트 중요**: system + tools가 매 요청 캐시됨. 큰 변경은 캐시 무효화 → 비용 증가. 점진적 개선이 캐시 친화적
- **Extended thinking 활용 극대화**: thinking budget 2000 토큰. v1에서는 명시적 사고 단계 가이드 추가를 보류했음. v2에서 다룰 만한 카드

## 9. 페르소나·대화 시작 제안

4. "`return_not_dev_term`과 `return_possible_typo` 분기의 **판정 기준**이 현재 프롬프트에서 충분히 명확한가요? 오판이 일어날 수 있는 회색 지대(예: `react`는 영어 일반어이자 라이브러리명)를 어떻게 다뤄야 하나요?"
5. "한 번에 제시할 수 있는 **최소 변경량으로 품질을 가장 크게 올릴** 단일 편집을 추천해주세요. 캐시 무효화 1회 비용을 정당화할 만한 것."

**대화 흐름 팁**
- 전문가가 제안한 수정안은 항상 **근거**를 물어볼 것 (왜 이 변경이 품질을 올리는지)
- 수정안이 나오면 **전·후 diff**를 명확히 받아서 실제 코드 반영하기 쉽게
- 비용 고려: 프롬프트가 길어지면 매 요청 캐시 크기가 커짐. 효과 대비 길이 늘리는 타당성 확인
- v1 항목 다섯 개와 겹치는 제안이 나오면 "이미 적용됨"으로 알려주고 다른 각도를 요청

## 10. 산출물 형식 — 반드시 md 파일로

대화의 마지막에는 전문가가 **`handoff-v2.md`** 파일 하나를 산출하도록 요청하세요. 이 파일을 그대로 Claude Code에 넘겨 코드에 반영합니다.

### 요청 문구 예시

> 지금까지 논의된 내용을 바탕으로, Claude Code가 그대로 코드에 반영할 수 있는 **`handoff-v2.md`** 핸드오프 문서를 작성해주세요. 형식은 첨부한 `handoff-v1.md`를 그대로 따르고, 다음을 포함하세요:
>
> 1. **변경 요약 표** (#, 변경 내용, 우선순위, 캐시 영향)
> 2. **각 변경별 상세**:
>    - 대상 파일 (경로)
>    - 위치 (몇 번째 섹션·블록 뒤·앞)
>    - 변경 전 → 변경 후 (정확한 텍스트, 들여쓰기 포함)
>    - 근거 (왜 이 변경이 품질을 올리는지)
>    - 예상 부작용 / 회피책
> 3. **반영 후 스모크 테스트**: 어떤 keyword로 어떤 결과를 확인할지
> 4. **반영 순서** (커밋 묶음 권장안 포함)
> 5. **명시적으로 하지 않는 것** (논의했으나 채택하지 않은 제안과 그 이유)
>
> 캐시 무효화 비용을 의식하여, system prompt 본문 수정은 가급적 한 묶음으로 통합해주세요.

### 핸드오프 문서가 갖춰야 할 품질 기준

- ✅ 변경 전·후 텍스트가 **그대로 복붙 가능**할 것 (요약/의역 금지)
- ✅ 대상 파일 경로가 명시될 것 (`DevEtym/DevEtym/Services/ClaudeAPIService.swift` 등)
- ✅ 각 변경마다 **근거 한 단락** 있을 것 (without why = 거부)
- ✅ sanity test / drift test 영향이 있으면 **함께 갱신 지시** 포함
- ✅ 캐시 무효화 횟수를 1회로 묶도록 **반영 순서** 권장
