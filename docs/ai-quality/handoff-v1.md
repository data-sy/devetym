# DevEtym 시스템 프롬프트 개정 — 핸드오프

> 프롬프트 엔지니어링 리뷰 결과를 바탕으로 한 구체적 변경 지시.
> Claude Code에게 이 문서 + 현재 시스템 프롬프트 파일 + sanity test 파일을 함께 주고 반영 요청.

---

## 변경 요약

| # | 변경 | 우선순위 | 캐시 영향 |
|---|---|---|---|
| 1 | Few-shot 예시 4번(`bug`) 추가 — 불확실한 어원/민간어원 처리 시연 | 필수 | 캐시 무효화 1회 |
| 2 | `namingReason` 길이 상한 `300` → `270` 조정 (프롬프트 + tool schema 양쪽) | 필수 | 위와 함께 묶음 |
| 3 | `aliases` 필드 스코프 규칙 추가 | 필수 | 위와 함께 묶음 |
| 4 | Sanity test에 `bug` keyword 추가 | 필수 (#1과 연동) | 해당 없음 |
| 5 | CI drift 체크 추가 — 프롬프트 본문과 tool schema description의 길이 기준 문자열 일치 검증 | 권장 | 해당 없음 |

**묶음 원칙**: 1~3은 시스템 프롬프트 문자열을 수정하므로 같은 릴리즈에 묶어 캐시 무효화 비용을 1회로 제한.
**명시적 비변경**: tool schema description 전면 축약은 근거 부족으로 **하지 않음**. 현재 중복 상태를 유지하되 drift만 방지.

## 변경 1: Few-shot 예시 4번 추가

### 대상 파일
현재 시스템 프롬프트 문자열이 정의된 파일.

### 위치
기존 예시 3(`daemon`) 블록 바로 뒤에 추가.

### 추가할 내용

```
[모범 답안 예시 4 — 어원이 불확실한 경우]
입력: bug
return_term_entry input:
{"keyword":"bug","aliases":["버그"],"category":"기타","summary":"프로그램의 의도치 않은 오작동이나 결함","etymology":"정확한 어원은 불분명하나, 영어 'bug'(벌레)의 비유적 용법에서 유래한 것으로 본다. 1878년 에디슨이 편지에서 기계 결함을 'bug'로 지칭한 기록이 있어, 소프트웨어 이전 엔지니어링 속어로 이미 쓰이고 있었다.","namingReason":"1947년 하버드 Mark II 컴퓨터의 릴레이에서 실제 나방이 발견되어 로그에 붙여진 'first actual case of bug being found'라는 일화가 유명하나, 이는 기존 은유를 문자 그대로 실행한 농담에 가깝고 용어의 기원 자체는 아니다. 기계 속을 기어다니며 예측 불가능한 고장을 일으키는 '벌레'라는 은유가 소프트웨어의 숨은 결함에 그대로 이식되어 오늘날 표준 용어로 자리 잡았다."}
```

### 추가 근거
- 기존 세 예시(`mutex`, `jpa`, `daemon`)는 모두 어원이 명확한 케이스로, "정확한 어원은 불분명하나" 트리거에 대한 demonstration이 없었음
- 부정 제약("민간어원을 사실처럼 단정하지 마세요")에 대한 긍정 예시 역할도 겸함
- Grace Hopper 나방 일화를 "농담에 가깝고 용어의 기원은 아니다"로 선긋는 방식이 민간어원 분별의 템플릿이 됨
- 범주 `기타`의 활용 예를 하나 더 확보 (기존: daemon 하나뿐)

### 예상 부작용
없음. 예시 추가는 기존 예시의 스타일을 침범하지 않음.

## 변경 2: `namingReason` 상한 조정

### 대상 파일
- 시스템 프롬프트 파일
- Tool schema 정의 파일 (`return_term_entry`)

### 변경 내용

**시스템 프롬프트 `[필드별 작성 기준]` 섹션**:

```
- namingReason: 150~300자.
```

→

```
- namingReason: 150~270자.
```

**Tool schema `namingReason` field description**:

```
150~300자 분량. 어원상 의미와 개발 현장에서의 실제 쓰임 사이에 다리를 놓는 설명.
```

→

```
150~270자 분량. 어원상 의미와 개발 현장에서의 실제 쓰임 사이에 다리를 놓는 설명.
```

### 근거
- 현재 샘플(`idempotent`)이 320자로 상한 +20 초과
- Sonnet 계열에서 범위의 상한이 target으로 작동하는 패턴 — 상한을 10~15% 낮춰 쓰는 회피책이 검증되어 있음
- 320자 샘플의 마지막 문장이 재진술에 가까워, 상한을 낮춰도 내용 손실 없음 (오히려 불필요한 마무리 문장이 잘림)

### 검증 방법
반영 후 다음 keyword로 실측:
- `idempotent` (원래 320자였음 — 270 이하로 떨어지는지)
- `mutex` (여전히 자연스러운지)
- `bug` (새 예시가 270 이하로 생성되는지)

### 기존 `etymology`에 대한 결정
`etymology`는 현재 상한 120자, `idempotent` 샘플 127자로 +7. 임팩트가 작아 **이번 릴리즈에서는 건드리지 않음**. 변경 1~3 반영 후 관찰.

## 변경 3: `aliases` 스코프 규칙 추가

### 문제
현재 프롬프트에는 `aliases`에 무엇을 담아야 하는지 명시적 규칙이 없음. 기존 예시들도 서로 다른 패턴:
- `mutex`: 한글 음차 + 풀네임
- `jpa`: 풀네임(영문) + 한글 풀네임
- `daemon`: 한글 음차 + 철자 변이

이 상태에서 모델이 "소프트웨어 결함" 같은 **정의/번역**을 alias로 넣을 위험이 있음 (예: bug의 초기 시안에 들어갈 뻔함).

### 대상 파일
시스템 프롬프트 파일.

### 위치
`[필드별 작성 기준 — return_term_entry]` 섹션, `summary` 항목 위에 새로 추가. (필드 순서대로 배치하기 위함. 기존 섹션은 summary → etymology → namingReason 순서만 다룸.)

### 추가할 내용

```
- aliases: 동일 개념을 지칭하는 대체 표기의 배열. 허용:
  (1) 한글 음차 (예: "뮤텍스", "데몬")
  (2) 약어의 풀네임 (예: "Java Persistence API")
  (3) 철자 변이 (예: "demon")
  정의·번역·상위 개념은 포함하지 않는다 (예: "소프트웨어 결함"은 alias가 아님).
  보통 1~3개. 적절한 대체 표기가 없으면 빈 배열을 반환한다.
```

### 근거
- 사용자가 검색창에 `"소프트웨어 결함"`을 넣고 `bug`를 기대한다는 보장이 없음 → UX 관점에서도 문제
- category enum과 동일한 성격: 모델 자유재량으로 두면 안 되는 스코프 질문
- 기존 세 예시와 새 예시(`bug`) 모두 이 규칙에 부합 → 예시 수정 불필요

## 변경 4: Sanity test 업데이트

### 대상 파일
`test_systemPrompt_containsFewShotExamples` 테스트를 포함한 파일.

### 변경 내용
현재 expected keyword list: `mutex`, `jpa|JPA`, `daemon`

→ `bug` 추가.

### 대소문자 처리
기존 `jpa|JPA` 패턴과 동일한 관대함은 `bug`에는 불필요 (프롬프트 내 표기가 소문자 고정). 단순 `bug` 매칭.

### 검증
테스트 실행 후 pass 확인. 예시 4가 누락되면 fail.

## 변경 5: CI drift 체크 (권장, 즉시 반영)

### 문제
시스템 프롬프트 본문의 길이 기준과 tool schema description의 길이 기준이 **중복**되어 있음. 한쪽만 수정하면 drift 발생. (이번 변경 2가 정확히 그 drift 방지를 수동으로 하는 중)

### 왜 축약하지 않나
Tool schema description을 "한 줄 요약"으로 줄이자는 기존 제안은 **A/B 근거가 부족**하여 철회. 일치하는 중복은 관찰된 실패 모드가 없고, 충돌 중복만이 문제. 현재 상태는 일치하는 중복.

### 대안: drift 감지 테스트
다음 규칙을 검증하는 간단한 테스트 추가:

- 시스템 프롬프트 본문에서 `summary: 20~30자` 추출
- Tool schema `summary` description에서 `20~30자` 확인
- 두 값이 일치하지 않으면 fail
- `etymology`, `namingReason`도 동일

### 구현 팁
정규식 한 줄씩이면 충분:
```
/summary: (\d+~\d+)자/
/etymology: (\d+~\d+)자/
/namingReason: (\d+~\d+)자/
```
양쪽에서 캡처한 값이 일치하는지 assert.

## 반영 후 스모크 테스트

실제 API로 다음 keyword를 쿼리하고 수작업 검증:

### 길이 준수
- `idempotent` — namingReason 270 이하인지
- `mutex` (리그레션) — 자연스러움 유지
- `cookie` — 새 예시 효과로 "정확한 어원은 불분명하나" 트리거되는지

### 민간어원 분별
- `bug` — Grace Hopper 일화를 사실로 단정하지 않는지
- `ping` — Mike Muuss 작명 / 소나 의성어 두 설이 균형 있게 다뤄지는지

### Aliases 스코프
- 임의의 용어 3~5개에 대해 aliases 필드에 정의·번역이 섞여 들어오지 않는지

### 리그레션 체크
- `jpa`, `daemon` — 기존 출력 대비 질 저하가 없는지

## 명시적으로 하지 않는 것

- **Tool schema description 전면 축약**: 근거 부족.
- **`etymology` 상한 조정**: 임팩트가 작아 관찰 후 판단.
- **Thinking 유도 블록 추가**: 변경 1의 효과를 먼저 관찰한 뒤 별개 릴리즈로.
- **Few-shot 예시 교체**: 기존 3개는 커버리지가 좋음. 추가만.
- **페르소나 문구 수정**: "사전 데이터 제공자"가 톤 요구와 부합. 건드리지 않음.
