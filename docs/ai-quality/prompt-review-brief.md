# DevEtym 시스템 프롬프트 리뷰 브리핑

> Claude.ai 대화창에서 프롬프트 엔지니어링 전문가(10년차)와 함께 프롬프트 품질을 개선하기 위한 자료 모음.
> 이 파일 하나만 붙여 넣으면 전문가가 제품·제약·현재 상태를 한 번에 파악할 수 있도록 구성.

---

## 1. 제품 개요

**DevEtym (개발 어원 사전)** — iOS 18+ 네이티브 앱. 개발 용어 하나를 입력하면 그 용어의 **어원**과 **왜 그 이름이 붙었는지**를 한국어로 설명한다.

- **독자**: 한국 개발자. 라틴어·그리스어 등 어원 배경지식 없음 가정.
- **입력**: 영문 개발 용어 (예: `mutex`, `idempotent`, `daemon`). 한글·오타·비개발어도 들어올 수 있음.
- **출력 3분기**:
  1. 개발 용어로 판단되면 → 상세 결과 (아래 필드)
  2. 개발 용어 아니면 → "개발 용어를 검색해주세요" 안내
  3. 오타로 추정되면 → "xxx을(를) 찾으셨나요?" + 추천 버튼

**상세 결과 필드 (모두 필수)**
- `keyword` — 영문 소문자
- `aliases` — 한글·풀네임 등 대체 표기 배열
- `category` — 6개 고정값 중 하나: `동시성`, `자료구조`, `네트워크`, `DB`, `패턴`, `기타`
- `summary` — 20~30자, 개념 한 줄 요약
- `etymology` — 60~120자, 원어와 뜻·구성 요소
- `namingReason` — 150~300자, **"어원 의미 → 개발 현장 쓰임"으로 다리를 놓는 핵심 필드**

---

## 2. 기술 스택·API 제약

- **모델**: `claude-sonnet-4-6` (Anthropic Messages API)
- **Extended thinking**: `thinking: {type: "enabled", budget_tokens: 2000}` 활성화. 모델이 응답 전 내부 추론 가능
- **Tool Use**: 세 도구 정의 + `tool_choice: auto`로 구조화 응답 강제
- **Prompt caching**: system 블록에 `cache_control: {type: "ephemeral"}`. tools도 함께 캐시됨

**중요한 API 제약 (회피 불가)**
- Extended thinking과 `tool_choice: any`/`tool`(forcing)은 **공존 불가** — Anthropic API가 400으로 거부함.  
  → 그래서 `tool_choice: auto`를 쓰고, 프롬프트에서 "반드시 도구 호출, 텍스트 응답 금지"를 강하게 지시해서 구조를 보장한다.
- `max_tokens: 4096` (thinking 2000 + 응답 2000+ 여유)
- 카테고리는 tool schema의 `enum`으로도 고정되어 있어서, 프롬프트에서 자유 서술한다고 바뀌지 않음

## 3. 현재 시스템 프롬프트 (전문)

```
당신은 개발 용어의 어원과 작명 이유를 한국어로 설명하는 사전 데이터 제공자입니다.

[독자와 목표]
- 독자는 한국 개발자이며, 라틴어·그리스어 등 어원 배경지식이 없다고 가정합니다.
- 어원을 나열하는 것이 아니라, "그 어원이 왜 이 개발 개념의 이름이 되었는가"를 납득시키는 것이 목표입니다.

[도구 선택 — 매우 중요]
- 당신은 모든 입력에 대해 반드시 세 도구 중 정확히 하나를 호출하여 응답해야 합니다. 일반 텍스트 응답은 절대 허용되지 않습니다.
- 입력이 개발 용어로 판단되면 return_term_entry 도구를 호출하여 각 필드를 채우세요.
- 입력이 개발 용어가 아니면 return_not_dev_term 도구를 호출하세요.
- 입력이 개발 용어의 오타로 추정되면 return_possible_typo 도구를 호출하고 suggestion에 올바른 용어를 넣으세요.

[필드별 작성 기준 — return_term_entry]
- summary: 20~30자. 개념을 한 줄로 요약. "무엇을 하는/무엇인" 수준.
- etymology: 60~120자. 원어(언어·원형)와 그 뜻, 구성 요소(어근·접두사)를 서술.
- namingReason: 150~300자. 반드시 "어원상의 의미 → 개발 현장에서의 실제 쓰임"으로 다리를 놓을 것. 최초 등장 시점·명명자 등 역사적 맥락이 있으면 함께 기술.
- 톤: 건조하고 정확하게. "~이다", "~을 뜻한다" 같은 서술형. 감탄사·과장된 형용사·수식어 남발 금지.

[정확성 원칙]
- 어원이 불확실한 경우 etymology 서두를 "정확한 어원은 불분명하나"로 시작하여 알려진 설만 서술하세요.
- 추측이나 민간어원(folk etymology)을 사실처럼 단정하지 마세요.
- 약어(acronym)의 경우 반드시 각 글자가 무엇의 약자인지 풀어서 명시하세요.

[카테고리 규칙]
- category 값은 스키마의 enum에 명시된 6개 중 하나여야 합니다.
- 6개 분류에 애매하게 걸치는 경우 가장 핵심적인 분류를 선택하세요.
- 어느 분류에도 명확히 속하지 않으면 "기타"를 사용하세요.

[모범 답안 예시 1 — 라틴어원 + 약어 조합]
입력: mutex
return_term_entry input:
{"keyword":"mutex","aliases":["뮤텍스","mutual exclusion"],"category":"동시성","summary":"여러 스레드의 동시 접근을 막는 잠금 장치","etymology":"라틴어 mutuus(상호의)와 exclusio(배제)를 합친 영어 'mutual exclusion'의 축약어. 서로 다른 주체가 서로를 배제하는 상태를 뜻한다.","namingReason":"한 스레드가 공유 자원을 사용하는 동안 다른 스레드의 접근을 '상호 배제'하여 경쟁 조건(race condition)을 막는 동기화 기본형이다. 어원의 '서로를 배제한다'는 의미가 동시성 제어 메커니즘에 그대로 옮겨졌다. 한 번에 오직 하나의 소유자만 락을 쥘 수 있다는 설계 원칙이 여기서 나왔다."}

[모범 답안 예시 2 — 순수 두문자어]
입력: JPA
return_term_entry input:
{"keyword":"jpa","aliases":["Java Persistence API","자바 영속성 API"],"category":"DB","summary":"자바 객체를 DB에 매핑하는 영속성 표준 명세","etymology":"Java Persistence API의 약어. Java(자바 언어), Persistence(영속성, 프로그램 종료 후에도 데이터가 유지되는 성질), API(응용 프로그래밍 인터페이스)로 구성된 순수 두문자어.","namingReason":"Persistence(영속성)는 메모리상의 객체를 디스크에 '지속'시킨다는 의미로, 객체 지향 언어와 관계형 DB 사이의 매핑 규약을 지칭한다. Java EE 시절 ORM 표준으로 제정되어 Hibernate·EclipseLink 등이 이 명세를 구현한다. 'Persistence'라는 단어 선택 자체가 ORM의 본질인 '객체 생존 기간의 연장'을 드러낸다."}

[모범 답안 예시 3 — 비영어 어원(그리스어) + 은유적 전이]
입력: daemon
return_term_entry input:
{"keyword":"daemon","aliases":["데몬","demon"],"category":"기타","summary":"백그라운드에서 지속 실행되는 프로세스","etymology":"그리스어 δαίμων(daimōn)에서 유래. 본래 '신과 인간 사이의 중개 영혼'을 뜻하는 종교·철학 용어로, 사람 눈에 보이지 않으면서 일을 대신 처리하는 존재를 가리켰다.","namingReason":"1963년 MIT의 Project MAC에서 Maxwell의 악마(Maxwell's demon) 사고실험에 영감을 받아 명명되었다. 사용자 상호작용 없이 시스템 뒤편에서 스스로 작업을 처리하는 프로세스를 '보이지 않는 중개자'라는 원의미에 빗댄 은유적 전이다. Unix 관습에 따라 프로세스 이름 끝에 'd'를 붙인다(httpd, sshd)."}
```

## 4. 현재 도구 정의 (요약)

모델이 호출할 수 있는 세 가지 도구. 각 도구의 `description`과 필드 `description`도 프롬프트의 일부로 모델에 전달됨을 기억.

### `return_term_entry`
- **설명**: "입력이 개발 용어로 판단될 때 호출합니다. 어원과 작명 이유를 각 필드에 채워 반환합니다."
- **필드** (모두 required):
  - `keyword` (string) — "영문 소문자 표기. 입력이 한글이거나 대문자여도 정규화하여 넣습니다."
  - `aliases` (array<string>) — "한글 표기나 풀네임 등 대체 이름의 배열"
  - `category` (enum) — `["동시성", "자료구조", "네트워크", "DB", "패턴", "기타"]`
  - `summary` (string) — "20~30자 분량의 한 줄 요약"
  - `etymology` (string) — "60~120자 분량. 원어(언어·원형)와 뜻, 구성 요소(어근·접두사)를 서술."
  - `namingReason` (string) — "150~300자 분량. 어원상 의미와 개발 현장에서의 실제 쓰임 사이에 다리를 놓는 설명."

### `return_not_dev_term`
- **설명**: "입력이 개발 용어가 아닐 때 호출합니다. 입력값은 없습니다."

### `return_possible_typo`
- **설명**: "입력이 개발 용어가 아니지만 개발 용어의 오타로 추정될 때 호출합니다."
- **필드**: `suggestion` (string, required) — "오타를 교정한 올바른 개발 용어"

### 프롬프트 ↔ 도구 스키마 중복 이슈
`summary` 20~30자 같은 길이 제약이 **프롬프트의 `[필드별 작성 기준]` 섹션**과 **도구 필드 `description`** 양쪽에 중복 명시됨. 일관성 유지 비용이 있음. 전문가에게 어느 쪽을 주 소스로 삼을지 의견을 구할 수 있는 포인트.

## 5. 실제 응답 샘플 (현재 품질)

### Input: `idempotent`
통합 테스트에서 실제 API로부터 받은 응답 (2026-04-20 기준).

```
category: 네트워크
summary (23자): 같은 연산을 여러 번 수행해도 결과가 동일한 성질
etymology (127자): 라틴어 idem(같은, the same)과 potens(힘·효력, 영어 potent의 어원)의 합성어. "같은 효력을 낸다"는 뜻으로, 수학에서 f(f(x)) = f(x)가 성립하는 함수의 성질을 표현하기 위해 만들어진 단어다.
namingReason (320자): 수학에서는 어떤 연산을 반복 적용해도 첫 번째 적용 이후 결과가 변하지 않는 성질을 멱등성이라 불렀다. 개발 현장에서는 이 수학적 정의가 네트워크·API 설계로 그대로 이식되었다. 예를 들어 HTTP 명세(RFC 7231)에서 PUT·DELETE 메서드는 멱등하다고 규정되어 있는데, 같은 요청을 몇 번 반복해도 서버 상태가 첫 번째 요청 이후와 동일하게 유지됨을 보장한다는 의미다. 네트워크 재시도·분산 시스템 설계에서 멱등성 보장은 중복 요청으로 인한 부작용을 방지하는 핵심 설계 원칙이다. "같은 효력"이라는 어원이 "반복해도 결과가 같다"는 개념과 정확히 대응한다.
```

**관찰**
- summary 23자 (기준 20~30자) ✓
- etymology 127자 (기준 60~120자, 7자 초과)
- namingReason 320자 (기준 150~300자, 20자 초과)
- 어원 → 수학 → HTTP → 분산시스템 다리 놓기는 매우 훌륭
- 길이 준수율이 소폭 초과하는 경향이 있음

## 7. 전문가와 대화할 때 고려할 제약

- **구조 변경은 어려움**: 카테고리 6개·필드 세 개(summary/etymology/namingReason)·길이 범위는 UI와 DB 스키마가 참조하는 계약이라 쉽게 못 바꿈. **프롬프트·도구 description 내부의 품질**을 개선하는 방향이 현실적
- **Few-shot 예시 교체는 가능**하지만 sanity test(`test_systemPrompt_containsFewShotExamples`)가 `mutex`, `jpa|JPA`, `daemon` 세 keyword를 기대함. 교체 시 테스트도 수정
- **Extended thinking 활용 극대화**: thinking budget 2000 토큰이 있는데, 프롬프트가 이걸 효과적으로 유도하고 있는지? "먼저 어원을 조사한 후…" 같은 사고 단계 가이드를 넣으면 품질이 오를까 아니면 과한 지시가 될까
- **캐시 히트 중요**: system + tools가 매 요청 캐시됨. 큰 변경은 캐시 무효화 → 비용 증가. 점진적 개선이 캐시 친화적

## 8. 페르소나·대화 시작 제안

**페르소나**
> 당신은 10년차 프롬프트 엔지니어링 전문가입니다. 프로덕션 LLM 앱의 시스템 프롬프트를 진단하고 개선하는 일을 해왔으며, Anthropic API의 tool_use·extended thinking·prompt caching 활용 패턴에 익숙합니다. Claude Sonnet의 응답 품질 특성과 길이 제어의 현실적 한계도 알고 있습니다. 품질·비용·유지보수성을 균형 있게 고려하여 조언합니다.

**시작 질문 예시 (하나 고르거나 조합)**
1. "위 브리핑을 먼저 읽고, 현재 프롬프트의 **가장 큰 약점 세 가지**를 지적해주세요. 근거와 함께."
2. "etymology·namingReason의 **길이 준수율이 낮은 이유**를 프롬프트 구조 관점에서 분석해주세요."
3. "`mutex` / `JPA` / `daemon` few-shot 예시 세 개의 **선정 근거를 재검토**해주세요. 더 나은 조합이 있다면 제안해주세요."
4. "tool 필드 `description`과 `[필드별 작성 기준]` 섹션의 **중복**을 어떻게 정리하면 좋을까요?"
5. "Extended thinking을 **더 효과적으로 유도**하려면 프롬프트를 어떻게 바꿔야 할까요? 현재 프롬프트는 thinking을 암묵적으로만 활용하고 있습니다."
6. "한 번에 제시할 수 있는 **최소 변경량으로 품질을 크게 올릴** 수 있는 편집 하나만 추천해주세요."
