# ADR 0008: 이슈 트래킹 — GitHub Issues + Projects

## Status
Accepted (2026-07-15)

## Context

출시 준비 과정에서 발견되는 오류·개선점을 모아 관리할 체계가 없었다(발견 즉시 수정하거나 handoff 문서에 산발 기록). 트래커 선정 기준은 단일: **현업에서 사용하는 패턴을 미리 경험한다** — 이슈 등록 → 브랜치 → PR(`Fixes #N`) → 자동 close로 이어지는 표준 워크플로를 실전 근육으로 만드는 것이 목적.

## Decision

**GitHub Issues를 이슈 트래커 정본으로 쓴다.** 칸반/진행 보드가 필요하면 별도 HTML 시각화를 만들지 않고 **GitHub Projects**(내장)를 쓴다.

운영 방식:
- 발견한 버그·개선점은 이슈로 등록 (`bug`·`enhancement` 등 라벨).
- 수정 브랜치에서 작업하고 PR 본문에 `Fixes #N`으로 연결 → 머지 시 이슈 자동 close.
- 이슈는 시점성 상태 추적용, ADR·specs는 결정·설계 정본 — 역할 분리 유지.

## Consequences

### Positive
- repo(원격 이미 존재)에 바로 시작, 별도 도구·계정 불필요. private repo에서도 동작.
- 이슈-브랜치-PR 연결 고리가 오픈소스 컨트리뷰트 워크플로와 동일 — 목적(현업 패턴 연습)에 직결.
- `gh` CLI로 등록·조회·close 전부 가능 → AI 협업 세션에서도 이슈를 읽고 걸고 닫을 수 있음.

### Negative
- 스프린트·에픽 계층 같은 정식 애자일 프로세스는 경험 못 함 — 그 연습이 필요해지면 Jira/Linear 병행 검토(별도 결정).

### Neutral
- 진행 상태의 정본이 handoff 문서에서 이슈 트래커로 일부 이동 — 다단계 작업 원장은 여전히 handoff, 개별 버그·개선점은 이슈.

## Alternatives Considered

1. **Jira** — 기각(보류). 국내 현업 표준이나 1인 프로젝트에 관리 오버헤드 과대. 스프린트 운영 연습이 필요해지는 시점에 재검토.
2. **Linear** — 기각(보류). 스타트업 현업에서 인기이나 Jira와 같은 이유로 보류.
3. **Notion** — 기각. 현업에서의 역할은 위키·기획 문서이지 이슈 트래커가 아님 — 선정 기준(현업 패턴)에 부적합.
4. **로컬 `issues/` 폴더 + md + HTML 보드** — 기각. 이슈-PR 자동 연결·알림이 없고 보드를 직접 제작해야 함 — 연습이 아니라 도구 제작이 됨. 현업 패턴 아님.
