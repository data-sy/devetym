## 1. 비용 지도 — 돈이 어디서, 얼마나 나가는가

모든 비용은 **Claude 모델 `claude-sonnet-4-6`** 한 종류에서 발생한다 (요금: 입력 100만 토큰당 $3, 출력 100만 토큰당 $15. 캐시된 입력은 약 1/10 가격).

### 표면 ① 개발 스크립트 (내 로컬, `ANTHROPIC_API_KEY` 직접 사용)

| 스크립트 | 하는 일 | 1회 실행 예상 비용 | 빈도 |
|---|---|---|---|
| `Scripts/generate_db.py` | 번들 DB 배치 생성 (10개 단위 배치, 실시간 API) | 배치당 ~$0.1 → 100개 용어 ≈ **$1 안팎** | 현재 deprecated 예정 — chat 탭 방식으로 대체됨 |
| `Scripts/prompt-probe/probe_prompt.py` | 프롬프트 품질 측정 (8셀 × 15키워드 = **120호출**) | 1회 실행 ≈ **$3~6** (주석의 "$1-2" 추정은 과소일 수 있음 — 실측 없음) | 프롬프트 개정 때만 |
| `Scripts/db-expand/api_sample.py` | chat↔API 일관성 검증 (라운드당 5~10호출) | **$0.1~0.5** | DB 확장 라운드마다 |

- **지금까지의 누적**: 번들 650개 전부 API로 만들었다면 ≈ $5~10 규모. 단, DB 확장은 이미 claude.ai 채팅 탭(구독 정액)으로 전환돼(spec "확정 결정") API 비용이 크게 줄어든 상태.
- **최악 시나리오**: ① probe를 습관적으로 재실행 (회당 $3~6이 눈에 안 보이게 누적) ② keywords 파일에 실수로 수백 개를 넣고 generate_db 실행 ③ **가장 근본적인 위험 = 가시성 0** — 스크립트 어디에도 `response.usage`(토큰 수)를 비용으로 환산해 남기는 코드가 없어, "이번 달 얼마 썼는지"를 알 방법이 Console 수동 확인뿐.
- 다행히 폭주 방어는 이미 어느 정도 돼 있음: generate_db는 JSON 실패 재시도 1회 제한, probe는 재시도 3회 상한 — **무한 재시도 루프 위험은 낮다**.

### 표면 ② 런타임 앱 (앱 → devetym-proxy → Claude)

- 호출 1건 비용: 입력 ~4천 토큰(시스템 프롬프트+도구 스키마, 대부분 캐시 히트 시 1/10 가격) + 출력 ~1~2천 토큰(thinking 2000 포함) ≈ **건당 $0.02~0.04**.
- 구조적 방어(이미 설계됨): 기기당 일일 한도(429) + ADR-0006 D1 read-through 캐시(용어당 1회 생성 후 동결) + 번들 승격 플라이휠(INV-12) → **같은 용어는 두 번 다시 과금되지 않는 구조**. 소규모 사용자 기준 정상 트래픽 비용은 **월 수 달러 이하**로 예상.
- **최악 시나리오 (이 표면의 진짜 위험)**:
  1. **프록시 본문 위조** — 현행 계약(ADR-0004 관찰)상 *클라이언트가 Anthropic 요청 본문 전체(모델·max_tokens·프롬프트)를 구성하고 프록시는 그대로 전달*한다. 프록시 URL은 앱에 박혀 있어 공개나 다름없으므로, 악의적 사용자가 앱 밖에서 프록시에 `max_tokens: 128000`·비싼 모델의 임의 요청을 보낼 수 있다. **내 키로 남의 챗봇을 돌려주는 구멍**.
  2. **기기 한도 우회** — `X-Device-Id`는 클라이언트가 자율 신고하는 헤더라, UUID를 바꿔가며 보내면 일일 한도가 무력화된다. 기기당 한도만으로는 전체 지출이 안 잡힘 → **전역(모든 기기 합산) 캡 부재**.
  3. **캐시 미스 남용** — 무작위 문자열을 계속 검색하면 매번 캐시 미스 = 매번 생성 비용. INV-10의 "프로그래밍 용어처럼 보이는가" 휴리스틱이 설계엔 있으나 아직 미구현.
- 극단값 감각: 위 구멍으로 하루 1,000건 미스가 발생하면 **하루 $30~50** — 한 달이면 세 자리 달러. 이것이 상한 층이 필요한 이유.

### 표면 ③ 향후 운영

- 사용자 증가 → 신규 용어 검색(캐시 미스)만 비용. 캐시+번들 구조 덕에 **비용이 사용자 수에 비례하지 않고 "새 용어 수"에 비례** — 시간이 갈수록 단조 감소(INV-12).
- DB 확장 라운드: chat 탭 정액 방식이라 API 비용은 검증 샘플(회당 <$1)뿐. Phase 7 자동화 진입 시 API 전환하면 라운드당 비용 재등장 → 그때 Batch API(50% 할인) 필수.
- 프롬프트 버전업 → 선택적 무효화 → 재생성: INV-5(gated)로 폭주는 설계상 차단. 다만 전체 재생성(650개+)을 결정하면 1회성 $10~20 — 이때도 Batch API로 반값.

## 2. 근본 구조 제안 — "계정을 나누고, 관문에서 재고, 정산은 Anthropic에 맡긴다"

국소 패치(스크립트마다 로그 한 줄)를 배제하고, 비용이 발생하는 **모든** 지점을 두 개의 장치로 덮는다:

```
                       ┌─ 워크스페이스 "dev-scripts"  ← 스크립트 전용 키
  Anthropic 조직 ──────┤
   (Admin 키로           └─ 워크스페이스 "proxy-runtime" ← 프록시 전용 키
    비용 자동 조회)
                                      ▲
                                      │ 유일한 런타임 관문
                              devetym-proxy
                         (모델·토큰 서버 강제 + 전역 캡 + usage 기록)
```

- **원리 1 — 표면별 계정 분리**: Anthropic Console의 **워크스페이스**를 2개로 나누고(개발 스크립트용 / 프록시 런타임용) 각각 전용 API 키를 쓴다. 그러면 "어느 표면이 얼마 썼는가"를 **코드 한 줄 안 고치고** Anthropic이 대신 나눠 계산해준다. 워크스페이스마다 **별도 spend limit(지출 상한)** 도 걸 수 있다. 이는 INV-10("기존 spend 상한·워크스페이스 분리를 재사용")과도 정합.
- **원리 2 — 정산은 Admin API에 위임**: Admin 키(`sk-ant-admin...`)로 조직 비용 리포트를 주기 조회하는 작은 스크립트 하나(`Scripts/cost/report.py`)가 **모든 표면의 비용 정본**이 된다. 스크립트 5개에 로깅을 심는 대신 조회 지점 1개.
- **원리 3 — 런타임 세부는 관문(프록시)에서**: 프록시는 런타임 비용이 지나가는 유일한 길목이므로, 여기 한 번 심으면 캐시 히트율·미스당 토큰·기기별 사용량이 전부 잡힌다. 앱 코드는 무변경(ADR-0006 "클라에 투명" 유지).

### 층별 옵션 비교와 추천

#### 가시성 층 (얼마 쓰는지 보이게)

| 옵션 | 비용 | 구현 난이도 | 유지보수 | 판정 |
|---|---|---|---|---|
| **A. 워크스페이스 분리 + Admin API 비용 리포트 스크립트** | $0 (Admin API 무료) | 낮음 (Console 클릭 + 스크립트 1개) | 거의 없음 | ✅ **추천 — 정본** |
| B. 프록시에 usage 기록 (D1 테이블) | $0 수준 (D1 무료 한도 내) | 중간 (프록시 수정) | 낮음 | ✅ **추천 — A의 보완** (히트율·기기별 세부는 A가 못 봄) |
| C. 스크립트마다 usage 로깅 공통 모듈 | $0 | 중간 (스크립트 5개 수정) | 스크립트 늘 때마다 반복 | ❌ 기각 — A가 커버하는 것을 국소 패치로 중복. 단 probe의 기존 cache 토큰 표시는 그대로 둠 |

#### 상한 층 (폭주해도 여기서 멈추게) — 3중 방벽, 전부 채택 권장

| 방벽 | 무엇을 막나 | 구현 | 판정 |
|---|---|---|---|
| **1. Console spend limit + 알림** (워크스페이스별) | 모든 폭주의 최종 안전망 — 어떤 구멍이든 여기서 끝 | 사람 손 5분 (코드 없음) | ✅ 지금 당장. 예산 미정이므로 **임시값: 조직 월 $25** (dev $15 / runtime $10) — 1~2개월 실측 후 조정 |
| **2. 프록시 하드닝** ① 요청 본문 서버 강제(모델·max_tokens·thinking을 프록시가 덮어씀 — 위조 무력화) ② 전역 일일 미스 캡(예: 하루 200건 초과 시 429) | 표면 ②의 최악 시나리오 1·2번 | 프록시 수정 (범위 포함 확인됨) | ✅ 추천 — D1 캐시 구축(CMP M3)과 같은 파일을 만지므로 **함께 구현이 효율적** |
| 3. 스크립트 예산 가드 (실행 전 비용 추정 출력 + 확인) | 표면 ①의 실수 실행 | generate_db는 deprecated 예정이라 probe에만 | ⚠️ 저우선 — 방벽 1이 이미 하드 캡. "예상 비용 $X — 계속?(y/n)" 프롬프트 정도만 |

| 항목 | 절감 | 판정 |
|---|---|---|
| **D1 캐시 + 번들 승격 로드맵 완주** (이미 설계됨) | 구조적 — 반복 비용을 0으로 | ✅ 최우선. 새로 만들 것 없음, ROADMAP대로 |
| Prompt caching | 이미 3표면 전부 적용돼 있음 (`cache_control: ephemeral`) | ✅ 유지 — 추가 작업 없음 |
| Batch API 전환 (50% 할인) | 배치 작업 반값 | ⏸️ 지금은 대상 없음 (DB 확장이 chat 정액으로 이동). **Phase 7 자동화 또는 전체 재생성 시 필수 채택** — 그때 결정 지점에 이 문서를 참조 |
| 모델 다운그레이드 (Haiku 4.5: 입력 $1/출력 $5) | 호출당 ~70% | ❌ 보류 — Sonnet 4.6은 품질 검증(v2 acceptance)을 통과한 자산. 어원 사실성이 제품 핵심이라 품질 리스크 > 절감액. 비용 데이터가 쌓여 런타임 비용이 실제 문제가 되면 재검토 |

## 3. 채택 결정 (요약)

1. **워크스페이스 2분할** (`dev-scripts` / `proxy-runtime`) + 키 재발급 — 표면별 비용 자동 분리.
2. **Console spend limit**: 임시 월 $25 (워크스페이스별 세분) + 50%/75% 이메일 알림. 실측 2개월 후 재조정.
3. **Admin API 비용 리포트 스크립트** `Scripts/cost/report.py` — Admin 키로 조직/워크스페이스별 비용을 조회해 "이번 달 표면별 지출" 표를 출력. 주 1회 손으로 실행(소규모에선 cron 불필요).
4. **프록시 하드닝** (devetym-proxy repo): 요청 본문 서버 강제 + 전역 일일 캡 + usage 기록(D1) — CMP M3 서버 작업과 병합 구현.
5. Batch API·모델 재검토는 **트리거 조건부 보류** (§2 표 참조).

## 4. 이 체계가 "근본적"인 이유 (한 문단)

비용이 발생할 수 있는 경로는 ①내 키를 든 로컬 스크립트 ②프록시 키를 든 런타임, 딱 두 갈래다. 이 체계는 두 갈래를 각각 **계정 수준(워크스페이스+spend limit)** 에서 감싸므로, 앞으로 스크립트가 새로 생기든, 프록시에 기능이 붙든, 내가 로깅을 깜빡하든 **자동으로 측정·상한 안에 들어온다**. 코드에 심는 것은 런타임 세부(히트율)를 보기 위한 프록시 한 지점뿐이다.
