> **작성**: 2026-07-14 (iOS 심사 제출 직후, 사람 발의) · **작업 브랜치**: `chore/doc-pruning` · **진입**: 새 세션에서 **"docs/handoff/26-07-14-doc-pruning-backlog.md 읽고 서류 정돈 진행하자"**.

## 0. 철학 (사람 결정 — 이 트랙의 헌장)

**미래의 것이 YAGNI이듯 과거의 것도 굳이 다 트래킹할 필요 없다.** M0→제출까지 쌓인 작업 문서(지그·원장·경위 서사)는 임무를 다했다. git 이력이 전 과정을 보존하므로 **삭제는 손실이 아니다** — 문서를 지워도 `git log`로 언제든 복구·열람 가능.

- **버릴 건 과감하게 버린다.** 완료된 지그·프리플라이트·캡처 레시피·검증 원장 등 "실행이 끝난 절차 문서"가 1순위.
- **바뀐 흐름(경위 서사)도 날린다.** "구안 → 폐기 → 개정 → 추기" 같은 결정 여정 서술은 삭제하고, **최종적으로 뭘 선택했는지 결과만 자기완결적으로** 남긴다. (예: 취소선 블록, 〔추기〕 체인, superseded 각주)
- **남기는 문서의 기준 = cold-start 자기완결성.** 맥락 없이 열어도 현재 상태·최종 결정·다음 행동이 바로 읽히는가.

## 2. 안전선

- **거버넌스·데이터면(specs·ADR·architecture·페르소나·prd·design)은 삭제/수정 금지** — stale 발견 시 제안+승인.
- 삭제는 반드시 **git 커밋으로**(이력 복구 가능성이 이 트랙의 전제). 브랜치 삭제 금지 규율 유지.
- **push·main 병합 = 사람 지시로만.** 작업은 `chore/doc-pruning` 브랜치에서.
- 미완 작업은 조용히 지우지 않는다 — 삭제 문서에 미완 항목이 박혀 있으면 ROADMAP/이 백로그로 **추출 후** 삭제.
- 심사 결과(승인/리젝) 대응이 이 트랙보다 우선 — 리젝 시 정돈 중단하고 E 핸드오프 §3-5.

## 3. 절차 스케치 (착수 세션용)

1. 인벤토리 실측(파일 트리 대조) → §1 표 확정.
2. 배치 단위로 [삭제]/[압축] 실행 — 압축은 "최종 결정·현재 상태·다음 행동"만 남기는 재작성.
3. 각 배치 커밋 → 전체 완료 시 사람에게 병합 요청.
4. 완료 오라클: **cold-start 테스트** — 정돈 후 남은 문서만 읽고 "현재 상태로 복귀 가능한가"를 새 눈으로 검증.
