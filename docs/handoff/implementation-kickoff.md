## 4. 멀티에이전트 자동화 계획

목표: 이 앱 구현을 멀티에이전트로 자동화하되, **마일스톤 경계마다 사람이 비준**한다(무인 end-to-end 아님 — 위 no-push 규율과 같은 정신).

### 파이프라인 (마일스톤 단위)
```
스펙 비준 하네스  →  구현 하네스  →  검증(green)  →  적대적 리뷰  →  [사람 비준]  →  다음 마일스톤
```
- **스펙 비준 하네스** — 해당 마일스톤의 spec 슬라이스 + architecture + ADR을 적대적으로 점검(완결성·모호성·테스트 가능성·정합성). 산출: 비준 or 갭 목록. **툴체인 불필요(순수 추론)라 리스크 낮음** — 언제든 돌려도 안전. (하이트래픽의 스펙 비준 하네스 + `design-review` 스킬 재사용/적응.)
- **구현 하네스** — 비준된 spec 슬라이스로 코드+테스트 구현, **빌드+테스트 green에 수렴**, 적대적 코드 리뷰. (하이트래픽의 구현 하네스 재사용/적응.)

### ⚠️ 핵심 통찰 — 검증 기반(substrate)이 먼저
- **스펙 비준 하네스는 안전, 구현 하네스는 검증 루프만큼만 믿을 수 있다.** 코틀린 코드 생성은 쉽지만 "Android·iOS 양쪽에서 컴파일되나"는 실제 툴체인이 있어야 안다. 없으면 에이전트는 *그럴듯하지만 미검증*인 코드를 낸다.
- 따라서 **M0의 진짜 산출물은 "빈 앱 실행"이 아니라 "에이전트가 호출 가능한 신뢰할 green 루프"**다 — `./gradlew :shared:test`, Android assemble, iOS 빌드가 도는 상태.
- **권고 시퀀스:**
  - **M0 = 사람 게이트(반자동).** 툴체인·버전 정렬·SKIE/Swift Export 결정·양 플랫폼 빌드까지. 불확실성·환경 의존이 가장 큰 단계라 무인 자동화하지 말 것. 여기서 green 루프를 세운다.
  - **M1~M8 = 하네스.** 각 마일스톤 파이프라인(비준→구현→green→리뷰). 마일스톤 간은 **순차**(의존), 마일스톤 내 독립 작업은 **병렬**(예: M6 화면들, M8 아이콘·스플래시 플랫폼별).

### 가드레일
- **버전 정렬을 사실로 확인** — Kotlin↔CMP↔플러그인 호환 조합은 KMP 최대 실패원(vol6 §7, [ADR-0001](../adr/0001-cross-platform-framework.md)). 에이전트가 **stale 버전을 하드코딩하지 않도록**, 착수 시 `klibs.io`/공식 문서로 확인시킨다.
- **에이전트에 정답을 박지 말 것** — 프롬프트엔 spec 슬라이스 + 제약 + 완료 기준(DoD)만 주고, 진단·설계·구현은 에이전트가 하게 한다(구체 코드/매핑을 미리 주입하지 않는다).
- **마일스톤 경계 사람 비준** 없이 다음으로 넘어가지 않는다. **푸시는 하네스가 절대 하지 않는다.**
- 네이밍은 젠더중립/여성형 기본(하네스·에이전트·패턴 명명 시).
- 다음 세션에서 워크플로(멀티에이전트) 실행은 **명시적 옵트인** 필요(그 세션에서 "멀티에이전트로 돌려줘" 류로 요청).

### 열어둔 질문
- 하이트래픽의 스펙 비준/구현 하네스가 어떤 형태(워크플로 스크립트 / 에이전트 정의 / 프롬프트 템플릿)인지에 맞춰, 이 repo용으로 이식/적응할지 결정. 다음 세션에서 그 하네스를 참조로 붙이면 됨.
