# DevEtym 서버측 캐시 & 콘텐츠 딜리버리 — 마일스톤

> ⚠️ **읽는 법 — 이 문서는 이제 별도 트랙이 아니라 CMP M1~M8에 빌트인이다.** 아래 M0~M7은 **캐시·딜리버리 고유 번호**이며 [`ROADMAP.md`](../ROADMAP.md)의 CMP 마일스톤(M0~M8)과 **번호가 겹쳐도 무관**하다. 별도 트랙으로 미루지 않고(=출시 후 없음), 처음부터 각 CMP 마일스톤 범위에 녹여 짓는다. 서버 코드는 별도 repo [`devetym-proxy`](https://github.com/data-sy/devetym-proxy)(read-through로 확장). 이 문서는 **불변식(§1 INV-1~12)·상세 스펙의 정본**이고, 실제 착수 순서·배치는 ROADMAP이 정본이다. 계약 결정은 [ADR-0006](adr/0006-server-cache-boundary.md).
>
> **캐시 번호 → CMP 마일스톤 매핑**:
>
> | 이 문서 | 내용 | CMP 배치 |
> |---|---|---|
> | M0 | term key 정규화·D1 스키마·entry 계약 | 클라 계약 = **CMP M1**(반영✅) / D1 = **CMP M3**(서버) |
> | M1 | Worker read-through 캐시 | **CMP M3**(서버) + **CMP M4**(클라 소비) |
> | M2 | single-flight(DO) | **CMP M3**(서버) |
> | M3 | 품질 게이트(validator/critic) | write-게이트 **CMP M3**(서버) / critic 승격 **CMP M8** |
> | M4 | 클라이언트 local-first pinning | 저장 **CMP M2** / 행위 **CMP M4** / UI **CMP M6** |
> | M5 | seed 번들 + 승격 잡 | **CMP M8** |
> | M6 | 콘텐츠 팩 백그라운드 동기화 | **CMP M8** |
> | M7 | 비용·남용 안전 + 무효화 (횡단) | **CMP M3**(서버, 횡단) |
>
> **문서 목적**: DevEtym의 AI 생성 어원 항목을 서버에 캐싱해 재사용하고, 릴리즈마다 앱 번들 커버리지를 늘려 Claude API 의존도를 점진 축소하는 작업의 마일스톤 정의.
>
> **역할 분담**
> - 이 문서 = **의도 + 확정 아키텍처(불변식)**. 재론 금지 제약.
> - Claude Code = 실제 코드베이스(Cloudflare Worker, D1, 클라이언트, db-expand 파이프라인)에 착지시켜 **각 마일스톤의 구현 스펙 작성**.
> - Claude Code는 이 문서 밖 맥락이 없다고 가정하고 self-contained하게 작성됨. 각 마일스톤의 "코드베이스 확인 필요" 항목이 Claude Code가 코드 지식으로 채울 지점.

## 1. 확정된 아키텍처 불변식 (NON-NEGOTIABLE)

스펙은 아래를 제약으로 전제한다. 이 결정들은 재론하지 않는다.

- **INV-1 · 3계층 read-through**: 조회 순서는 `클라이언트 로컬/번들 → 서버 D1 캐시 → Claude API`. API는 최후 수단이며, 미스 시 생성 결과를 **D1에 write-back**한다.
- **INV-2 · write-once + 멱등 read**: 용어당 정본은 1회 생성 후 동결. **read path에서 재생성 금지.** 두 번째 이후 요청은 저장된 것을 반환한다(새로 굽지 않는다).
- **INV-3 · term key 정규화**: `React` / `react` / `react.js`가 같은 키로 접히도록 canonical term key. 기존 alias 스키마를 재사용한다. 중복 항목이 생기면 안 된다.
- **INV-4 · 동시성**: 캐시가 빈 상태에서 동시 요청이 들어오는 레이스는 **single-flight**(term key당 코디네이터 1개)로 코얼레싱하고, 저장은 **first-write-wins**(term key unique 제약, `INSERT ... ON CONFLICT DO NOTHING`)로 처리한다. 늦게 생성된 중복본은 서버에서 버린다.
- **INV-5 · gated-newest-wins (raw LWW 금지)**: 재생성/교체는 오직 명시적 무효화(프롬프트·스키마 버전업, 또는 사용자 오류 신고)로만 트리거된다. 신규 후보는 품질 게이트 통과 시에만 정본을 교체하며, 구버전은 version 태그로 보존한다. 미검증 생성물이 검증된 정본을 덮어쓰지 않는다.
- **INV-6 · local-first pinning**: 사용자가 한 번 본 항목은 기기에 저장되며 그 사용자에게는 **불변**이다. 서버 정본이 이후 어떻게 진화하든 눈앞에서 바뀌지 않는다. 서버 진화는 "아직 그 용어를 안 본 사용자" 또는 "명시적으로 새로고침한 사용자"에게만 반영된다. 사전 항목은 전달되면 불변, 수정은 새 버전 추가.
- **INV-7 · 품질 게이트 분리**: `validator.py`(정량 룰) = **캐시 write 시점** 게이트(값싸고 결정적). `critic`(nuanced 4종) = **번들 승격 시점** 게이트(배치). 라이브 요청마다 critic을 돌리지 않는다. (기존 Phase 5 분리를 그대로 얹음.)
- **INV-8 · 생성 파라미터**: temperature 낮게(0–0.3). 정본을 원하므로 다양성을 소스에서 죽인다. best-of-N은 옵션이며 전 용어 기본값이 아니라 첫 등장 고가치 용어에만 선택 적용.
- **INV-9 · 버전 태깅**: 각 항목에 프롬프트/스키마 버전을 기록해 선택적 무효화·재생성이 가능하게 한다.
- **INV-10 · 비용/남용 안전선**: 기존 spend 상한·워크스페이스 분리를 재사용. 추가로 Worker에서 디바이스별 rate-limit + "프로그래밍 용어처럼 보이는가" 값싼 휴리스틱으로 비용 남용을 막는다.
- **INV-11 · 오프라인 우선 & 서버 온리 금지**: 사전 앱의 킬러 기능은 즉시·오프라인·지연 0. 서버가 죽어도 앱은 살아 있어야 한다. 서버 딜리버리는 SSOT가 아니라 **freshness(staleness 단축)**를 담당한다. authoring 정본은 db-expand 파이프라인 산출물이다.
- **INV-12 · head/tail 분리 + 승격 플라이휠**: 흔한·검증된 용어(head) = 구운 seed 번들. tail + 갓 생성분 = 서버 D1 캐시. 릴리즈마다 tail의 hot 항목을 critic 통과시켜 번들로 승격 → 번들 히트율 단조 증가.

## 2. 마일스톤

각 마일스톤은 **1개의 응집된 스펙 단위**. Claude Code는 마일스톤당 구현 스펙 1건을 산출한다.

### M0 — 데이터 기반: term key 정규화 + D1 스키마
- **목표**: 모든 것이 올라앉을 저장 substrate 확정.
- **스코프**: canonical term key 규칙, D1 스키마(entries / versions / aliases, term-key unique 인덱스, prompt·schema 버전 컬럼), 클라이언트에 반환되는 **entry 데이터 계약(shape)** 정의.
- **지켜야 할 불변식**: INV-3, INV-9.
- **스펙 산출물**: 스키마 DDL + 키 정규화 규칙 + 버전 컬럼 설계 + entry 응답 스키마.
- **의존성**: 없음(기반).
- **코드베이스 확인 필요**: 현재 D1 사용 여부/기존 스키마? 기존 alias 처리 로직 위치? 클라이언트가 이미 기대하는 entry shape?

### M1 — Worker read-through 캐시 경로
- **목표**: 핵심 비용 절감 루프.
- **스코프**: Worker 조회 순서(D1 → API), 미스 시 write-back, first-write-wins unique 제약 기반 멱등 write.
- **지켜야 할 불변식**: INV-1, INV-2, INV-4(저장 측), INV-8.
- **스펙 산출물**: Worker 요청 흐름 스펙 + 멱등 write 처리.
- **의존성**: M0.
- **코드베이스 확인 필요**: 현재 Worker 프록시 구조/라우팅? API 호출 래핑 위치?

### M2 — 동시성 하드닝 (single-flight)
- **목표**: 캐시 빈 상태 동시 요청 레이스 제거.
- **스코프**: term key당 Durable Object로 in-flight 생성 코얼레싱. 진행 중이면 후속 요청은 같은 결과를 대기 수신.
- **지켜야 할 불변식**: INV-4(코얼레싱 측).
- **스펙 산출물**: DO 설계 + 요청 합치기 프로토콜 + 타임아웃/실패 폴백.
- **의존성**: M1.
- **코드베이스 확인 필요**: 현재 Worker에 DO 사용 이력? wrangler 설정?

### M3 — 품질 게이트 배치
- **목표**: validator/critic를 캐시·승격 경계에 배선.
- **스코프**: `validator.py` = write 시점(통과분만 캐시 저장·서빙), `critic` = 승격 시점. validator 탈락 용어의 처리(저장 안 함? 재시도? 사용자 응답?) 정의.
- **지켜야 할 불변식**: INV-7, INV-5.
- **스펙 산출물**: 게이트 배치 스펙 + 실패 처리 흐름.
- **의존성**: M1 (write 경로 존재해야 함).
- **코드베이스 확인 필요**: validator.py 현재 실행 형태/인터페이스? Worker에서 호출 가능한가, 아니면 사전 검증 단계로 분리?

### M4 — 클라이언트 local-first pinning
- **목표**: 사용자가 본 항목의 불변 보장(INV-6). 서버 진화와 눈앞 안정성 분리.
- **스코프**: 기기 로컬 저장(현재 SwiftData / CMP 타깃), 본 항목 불변, 명시적 새로고침 경로.
- **지켜야 할 불변식**: INV-6, INV-11.
- **스펙 산출물**: 클라이언트 캐시 + pin/refresh 스펙.
- **의존성**: M0(entry 계약). 서버 트랙(M1–M3)과 **병렬 가능**.
- **코드베이스 확인 필요**: 현재 SwiftData 스키마? CMP 마이그레이션 진행 상황에 맞춰 어느 쪽에 구현?

### M5 — seed 번들 + 승격 잡
- **목표**: head/tail 분리와 릴리즈 승격 플라이휠.
- **스코프**: head/tail 분리 기준, 릴리즈 시 D1 hot 항목 export → critic 통과 → seed 번들에 접어넣기. "hot"의 선정 기준 정의.
- **지켜야 할 불변식**: INV-12, INV-7(critic 측).
- **스펙 산출물**: 승격 잡 스펙 + 선정 기준.
- **의존성**: M1(캐시에 데이터 축적) + M3(critic 게이트).
- **코드베이스 확인 필요**: 기존 db-expand 파이프라인 산출물 포맷? 현재 번들이 앱에 실리는 방식?

### M6 — 콘텐츠 팩 백그라운드 동기화
- **목표**: 콘텐츠 업데이트를 App Store 릴리즈에서 탈동조화하되 오프라인 유지.
- **스코프**: 버전드 콘텐츠 팩, delta/cursor 증분 동기화, 백그라운드 pull + 로컬 병합. 구운 seed로 시작 → 서버 델타를 당겨와 로컬 반영.
- **지켜야 할 불변식**: INV-11(오프라인 유지), INV-6(로컬 병합이 본 항목 불변 규칙과 충돌하지 않게).
- **스펙 산출물**: 팩 포맷 + 동기화 프로토콜(버전 커서, 증분) + 병합 규칙.
- **의존성**: M5(승격된 콘텐츠가 팩 소스). M4(로컬 저장소).
- **코드베이스 확인 필요**: 클라이언트 백그라운드 작업/네트워크 계층 현황?

### M7 — 비용·남용 안전 + 무효화 흐름 (cross-cutting)
- **목표**: 비용 남용 방지 + 재생성/무효화 파이프라인.
- **스코프**: 디바이스별 rate-limit, 용어 유효성 휴리스틱, 프롬프트 버전업 → 선택적 무효화 → gated 재생성 흐름.
- **지켜야 할 불변식**: INV-10, INV-5, INV-9.
- **스펙 산출물**: 안전 스펙 + 무효화/재생성 흐름.
- **의존성**: 횡단(M1·M3에 얹힘).
- **코드베이스 확인 필요**: 기존 spend 캡/워크스페이스 분리 설정 위치? 사용자 신고 UI 존재 여부?

## 3. 의존성 & 병렬 트랙

```
M0 ──► M1 ──► M2
        │
        └──► M3 ──► M5 ──► M6
M0 ──► M4 ───────────────► M6   (클라이언트 트랙, 서버 트랙과 병렬)
M7: 횡단 (M1·M3 위에 얹힘)
```

- **서버 트랙**: M0 → M1 → M2 / M3 → M5 → M6
- **클라이언트 트랙**: M0 → M4 → (M6에서 서버 트랙과 합류)
- **M7**은 M1·M3가 서면 언제든 붙일 수 있는 횡단 작업.

**권장 최소 출시 경로(MVP)**: M0 → M1 → M4. (write-once 캐시 + local-first만으로 "재사용으로 비용↓ + 본 항목 불변" 핵심 가치 확보. M2·M5·M6는 규모/수요 확인 후.)
